Develop an app

There are two ways to add actions to an app -- either though global functions or through a class with methods. We'll start with making some global functions.

1. Create a Python package in the apps directory with the name of the app you want to create.

In this example we will be recreating the some of the functions of the HelloWorld App in the apps directory, but we'll create them as global actions. We'll call our package HelloWorld. Inside this package we will create a file called actions.py which will contain our actions. The apps directory should look like this:


WALKOFF
|-- apps
  |-- __init__.py
  |-- HelloWorld
    |-- __init__.py
  |-- [Other apps]

The __init__.py file can be empty.

2. Write some functions

Each function you write in any module in this app can be turned into an action. We are going to start with the standard "Hello World" action, an action which takes an argument called "number" and increments it by one, an action which returns a specific letter of a string at a given index, and an action which prints a message to the console.


def hello_world(self):
    return self.introMessage

def returnPlusOne(self, number):
    return number + 1

def get_letter(self, string_in, index):
    return string_in[index]

def print_message(self, message):
    print(message)
    return 'success'

3. Convert the methods to actions

To mark a function as an action, you decorate the method with the action decorator from the apps package.


from apps import action

@action
def hello_world(self):
    return self.introMessage

@action
def returnPlusOne(self, number):
    return number + 1

@action
def get_letter(self, string_in, index):
    return string_in[index]

@action
def print_message(self, message):
    print(message)
    return 'success'

4. Create the api.yaml YAML metadata file to convert the methods to actions

For WALKOFF to recognize a function as an action, it must have an entry in the apps's api.yaml file, which must located in the app directory. The minimum required entry of the api.yaml is as follows:


walkoff: '0.1'
info:
  title: HelloWorld
  version: 1.0.0
actions:
  hello world:
    run: actions.hello_world
    returns:
      Success:
        schema:
          type: object
          properties:
            message:
              type: string
  plus one:
    run: actions.returnPlusOne
    parameters:
      - name: number
        type: number
        required: true
    returns:
      Success:
        schema:
          type: number
  get letter:
    run: actions.get_letter
    parameters:
      - name: string_in
        type: string
        required: true
      - name: index
        type: integer
        required: true
    returns:
      Success:
        schema:
          type: string
  print message:
    run: actions.print_message
    parameters:
      - name: message
        type: string
        required: true
    returns:
      Success:
        schema:
          type: string

Some things to notice about the API specification:

  1. The top-level name of the action does not necessarily match the "run" field of the action. This name will be displayed on web interface, so it can and should be human-readable. The "run" must be the same name as the method which describes the action.
  2. Each "run" field is prefixed with "actions." which reflects the module containing the actions. If we had other actions in a different module we would prefix the "run" field with that module's name. The "run" field omits the "apps.HelloWorld" portion of the full path to the action.
  3. The name of the parameter must match the name of the argument in the action.
  4. When your action is executed, inputs of different types will be cast into the appropriate types if possible, so there should be no reason to cast your arguments into the correct types in the action
  5. JSON schema descriptions such as limiting the length of the string or the maximum value of a number or integer are available to further validate the parameter. To learn more, look at App Api Schemas.
  6. It is good practice to return something from an action to help with debugging your action.
  7. The return types of the actions are not validated, only the inputs to the actions.
  8. It is good practice to return something from an action to help with debugging your action. If you wish to omit a return type, you do not need to specify any returns.
  9. The return types of the actions are not validated, only the inputs to the actions are validated.

We should give a little more information to help a user know more about what our app and its actions do. In particular, we should include descriptions on the action parameters. These will be rendered on the workflow editor and will help a user use your app better.


walkoff: '0.1'
info:
  title: HelloWorld
  version: 1.0.0
  description: A simple tutorial app
  contact:
    name: Walkoff Team
    email: some_email@company.com
  license:
    name: Creative Commons
actions:
  hello world:
    run: actions.hello_world
    description: Says hello
    returns:
      Success:
        description: A greeting message object
        schema:
          type: object
          properties:
            message:
              description: A greeting
              type: string
              enum: [Hello World]
  plus one:
    run: actions.returnPlusOne
    description: Adds one to a number
    parameters:
      - name: number
        description: The number to add to
        type: number
        required: true
        example: 5
    returns:
      Success:
        description: The incremented number
        schema:
          type: number
  get letter:
    run: actions.get_letter
    description: Gets a letter at a specified index from a string
    parameters:
      - name: string_in
        type: string
        required: true
        description: The string which will searched
        example: This is my string
      - name: index
        type: integer
        required: true
        description: The index of the letter
        example: 3
    returns:
      Success:
        description: The letter at the given position
        schema:
          type: string
          example: s
  print message:
    run: actions.print_message
    description: Prints a message to teh screen
    parameters:
      - name: message
        type: string
        required: true
        example: This is my message
    returns:
      Success:
        description: Completed successfully
        schema:
          type: string
          enum: [success]

Congratulations! You now have made your first WALKOFF app.

Go have a look in the webapp and see the name of your app included in the Workflow Editor and see the actions you defined included in the workflow editor. Next you'll learn how to use classes for your apps.
